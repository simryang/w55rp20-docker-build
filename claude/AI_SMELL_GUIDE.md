## AI 냄새란?

AI가 생성한 문서에서 나타나는 특유의 패턴으로, 다음과 같은 특징이 있습니다:
- 과도하게 친절하거나 설명적
- 불필요한 수식어 남발
- 기계적으로 느껴지는 구조화
- 메타 정보의 과도한 노출
- 비자연스러운 한국어 표현

**핵심 원칙**: 사람이 쓴 기술 문서처럼 간결하고 실용적으로 작성

---

## 주요 패턴과 개선 방법

### 1. 이모지 사용

❌ **나쁜 예**:
```markdown
✨ 주요 기능
🎯 빠른 시작
⚙️ 설정 방법
📚 문서
🔧 문제 해결
✅ 완료!
```

✅ **좋은 예**:
```markdown
## 주요 기능
## 빠른 시작
## 설정 방법
## 문서
## 문제 해결
완료했습니다.
```

**규칙**: 공개 문서에서 이모지 전면 금지 (내부 문서는 OK)

---

### 2. "~용" 표현

❌ **나쁜 예**:
- "Raspberry Pi Pico 기반 W55RP20 마이크로컨트롤러용 빌드 시스템"
- "초보자용 가이드"
- "개발자용 문서"

✅ **좋은 예**:
- "WIZnet W55RP20-S2E Docker 빌드 시스템"
- "입문자 가이드"
- "개발 문서"

**이유**: "~용"은 카탈로그나 마케팅 용어 느낌, 기술 문서는 명사형이 자연스러움

---

### 3. 과도한 수식어

❌ **나쁜 예**:
- "완전히 이해하고"
- "완벽하게 지원"
- "반드시 필요"
- "매우 간단"
- "누구나 쉽게"

✅ **좋은 예**:
- "이해하고"
- "지원"
- "필요"
- "간단"
- "사용할 수 있습니다"

**규칙**: 강조가 필요 없는 곳에 수식어 붙이지 않기

---

### 4. "~친화적" 표현

❌ **나쁜 예**:
- "초보자 친화적"
- "사용자 친화적"
- "개발자 친화적"

✅ **좋은 예**:
- "입문자용"
- "대화형"
- "개발자 지향"

**이유**: "~friendly"를 직역한 AI 특유의 표현

---

### 5. 감탄 표현

❌ **나쁜 예**:
- "끝! 이게 전부입니다."
- "여기부터 시작!"
- "간단하죠?"
- "완료입니다."
- "Docker 처음이라면 여기부터!"

✅ **좋은 예**:
- "완료했습니다." (필요시)
- "시작 지점"
- (간단하다는 표현 삭제)
- (완료 표현 삭제 - 문맥으로 충분)
- "Docker 입문 가이드"

**규칙**: 감정 표현 최소화, 사실만 전달

---

### 6. 과도한 보장 표현

❌ **나쁜 예**:
- "누구나 W55RP20 펌웨어를 빌드할 수 있습니다"
- "한 번만 실행하면 됩니다"
- "반드시 성공합니다"

✅ **좋은 예**:
- "W55RP20 펌웨어를 빌드할 수 있습니다"
- "한 번 실행하면 됩니다" 또는 "초기에만 필요합니다"
- "성공합니다"

**이유**: 과도한 보장은 비현실적이고 AI 특유의 패턴

---

### 7. 메타 정보 과다 노출

❌ **나쁜 예**:
- "BEGINNER_GUIDE.md - 완전 초보자 가이드 (832줄)"
- "build.sh (721줄)"
- "작성자: Claude (AI Assistant)"
- "Co-Authored-By: Claude Sonnet 4.5"
- "소요 시간: 약 20분"

✅ **좋은 예**:
- "BEGINNER_GUIDE.md - 입문자 가이드"
- "build.sh"
- (작성자 표시 삭제)
- (Co-Authored 삭제)
- "첫 실행: 약 20~25분 (Docker 이미지 빌드 포함)"

**규칙**:
- 줄 수는 개발자만 관심, 사용자는 무관
- AI 작성 여부는 내부 정보
- 레이블("소요 시간:") 제거, 내용만 표시

---

### 8. 기술 용어의 갑작스러운 사용

❌ **나쁜 예**:
```markdown
### 선택적 캐시 무효화 (REFRESH)
외부 리소스 업데이트 시:
```

✅ **좋은 예**:
```markdown
### 외부 리소스 업데이트
Docker 이미지는 한 번 받은 패키지와 SDK를 저장해서 재사용합니다.
보안 업데이트나 최신 버전이 필요할 때 특정 부분만 다시 받을 수 있습니다:
```

**규칙**:
- 기술 용어 사용 전 사용자 관점에서 먼저 설명
- 왜 필요한지 → 무엇인지 → 어떻게 하는지 순서

---

### 9. 전문 용어를 일반 문서에 노출

❌ **나쁜 예**:
```markdown
- **[claude/UX_DESIGN.md](claude/UX_DESIGN.md)** - UX 설계
  - 사용자 페르소나 분석
  - 대화형 플로우
  - UX 원칙
```

✅ **좋은 예**:
```markdown
- **[UX_DESIGN.md](claude/UX_DESIGN.md)** - UX 설계
  - 사용자 유형별 시나리오
  - 대화형 플로우
  - UX 원칙
```

**또는**: 내부 문서는 공개 문서에서 아예 언급하지 않기

**이유**: "페르소나 분석"은 UX 전문가 용어, 일반 사용자는 모름

---

### 10. 내부 구조 노출

❌ **나쁜 예**:
```markdown
└── claude/               # AI 협업 문서 (개발자용)
    ├── README.md
    ├── UX_DESIGN.md
    └── ...
```

✅ **좋은 예**:
```markdown
└── docs/                 # 문서
```

**규칙**: claude/ 같은 내부 폴더는 공개 문서에 언급 금지

---

### 11. 불필요한 나열

❌ **나쁜 예**:
- "간편, 권장"
- "빠름, 강력함, 편리함"

✅ **좋은 예**:
- "권장"
- "빠름"

**규칙**: 하나만 선택, 여러 개 나열은 AI 패턴

---

### 12. 섹션 설명의 과도한 열정

❌ **나쁜 예**:
```markdown

✅ **좋은 예**:
```markdown

  - 단계별 빌드 가이드
  - 문제 해결 체크리스트
```

**규칙**: 간결하게, 사실만, 감정 배제

## 감지 방법

### 자동 감지 스크립트

```bash
#!/bin/bash
# ai-smell-final.sh

echo "=== 최종 AI 냄새 측정 ==="
echo ""

echo "1. 이모지: $(grep -roh '[😀-🙏🌀-🗿🚀-🛿✅✓✗⚠⏱️]' README.md docs/*.md 2>/dev/null | wc -l)"
echo "2. AI 특유 표현: $(grep -ic '완벽\|쉽게\|끝!\|여기부터 시작' README.md docs/*.md 2>/dev/null | awk '{sum+=$1} END {print sum}')"
echo "3. 과도한 친화: $(grep -c '친화적' README.md docs/*.md 2>/dev/null | awk '{sum+=$1} END {print sum}')"
echo "4. Claude/AI 언급: $(grep -c 'Claude\|AI Assistant' README.md docs/*.md 2>/dev/null | awk '{sum+=$1} END {print sum}')"
echo "5. 작성자 표시: $(grep -c '작성자' README.md docs/*.md 2>/dev/null | awk '{sum+=$1} END {print sum}')"
```

### 수동 체크 포인트

1. **첫 문장 읽기**: 감탄사나 과도한 설명이 있는가?
2. **섹션 제목 확인**: 이모지나 "~용" 표현이 있는가?
3. **설명 톤 확인**: 과도하게 친절하거나 설명적인가?
4. **메타 정보 확인**: 줄 수, 작성자, 내부 폴더 노출이 있는가?
5. **기술 용어 확인**: 설명 없이 갑자기 등장하는가?

### 예제 3: 기술 용어 설명

❌ **Before (AI 냄새)**:
```markdown
### 선택적 캐시 무효화 (REFRESH)
외부 리소스 업데이트 시:

```bash
REFRESH="apt" ./build.sh
```
```

✅ **After (자연스러움)**:
```markdown
### 외부 리소스 업데이트
Docker 이미지는 한 번 받은 패키지와 SDK를 저장해서 재사용합니다.

```bash
REFRESH="apt" ./build.sh         # apt 패키지 다시 설치
```
```

**개선 포인트**:
- "선택적 캐시 무효화" → "외부 리소스 업데이트"
- 왜 필요한지 먼저 설명 (캐시 재사용 → 업데이트 필요성)
- 주석 개선 ("업데이트" → "다시 설치")

## 핵심 원칙 요약

1. **간결성**: 불필요한 수식어 제거
2. **사실성**: 감정 표현 최소화, 사실만 전달
3. **자연스러움**: 사람이 쓴 기술 문서처럼
4. **사용자 중심**: 기술 용어는 설명 후 사용
5. **메타 최소화**: 내부 정보 노출 금지

---

## 문서 작성 워크플로우

1. **초안 작성**: AI 도움으로 빠르게 작성
2. **AI 냄새 제거**: 이 가이드 참고하여 수정
3. **자동 검증**: `ai-smell-final.sh` 실행
4. **수동 검증**: 체크리스트 확인
5. **문서 검증**: `docs-validation.py` 실행
6. **최종 확인**: 전체 읽으면서 자연스러움 확인
